## SECURITY — NON-NEGOTIABLE RULES

**Never commit API keys or secrets to this repo.** The repo is public. Anything committed is permanently visible in git history.

- ✅ **Allowed in committed JS:** `SUPABASE_URL`, `SUPABASE_PUBLISHABLE_KEY` (anon key — designed to be public, protected by RLS)
- ❌ **Never in committed files:** Google Books key, ISBNdb key, DeepSeek key, any key with billing exposure, any service-role key
- All secret keys live **only** in Supabase Edge Function secrets (Dashboard → Edge Functions → Secrets)
- If a new external API is added and it requires a key, it **must** be proxied through an Edge Function — never called directly from browser JS with a key

**Before making the repo public (or keeping it public):** audit `js/supabase-config.js` and `js/main.js` for any value that isn't a publishable/anon key.

## Project State

BookSharez is a **community-first book marketplace**: a peer-to-peer used-book marketplace combined with a reader-identity system (per-user "Books I Have / Books I Want" shelves) and per-book discussion. The full product is defined in [docs/BOOKSHAREZ_PRODUCT_VISION.md](docs/BOOKSHAREZ_PRODUCT_VISION.md) (why/what/who) and [docs/BOOKSHAREZ_ARCHITECTURE.md](docs/BOOKSHAREZ_ARCHITECTURE.md) (full target design, phased). **Phase 1 ships only the marketplace foundation** — shelves, social graph, discussions, affiliate, and recommendations are later phases.

The repository contains:

1. **A vanilla HTML/CSS/JS app** ([index.html](index.html), [js/main.js](js/main.js), [css/style.css](css/style.css)) wired to **Supabase** (auth + Postgres). Working end-to-end and **verified live June 17**: real auth; browse/search reads active listings from the DB; selling persists a real listing with ISBN auto-fill and AI price suggestion (DeepSeek Edge Function); a dashboard with "Books I Have", "Books I Want", "For Sale", and "Profile" tabs; book detail page with photo gallery; condition filter + sort. No build step, no frontend framework. **Run with `node dev-server.js` → http://localhost:7654** — since July 4 the JS is native ES modules, which browsers refuse to load from `file://`, so the site must be served over http (any static server works; the verify harnesses expect port 7654).

2. **Spec + design docs** ([docs/](docs/)) — **vanilla HTML/CSS/JS + Supabase (Postgres, Auth, Storage, Edge Functions)** (June 12 ADR; **Next.js is only a post-validation graduation target**, see [docs/GRADUATION_CRITERIA.md](docs/GRADUATION_CRITERIA.md)). Schema, indexes, and RLS are live in Supabase ([db/schema.sql](db/schema.sql), RLS verified). See [ToDo.md](ToDo.md) for next steps.

This **is** a git repository (initialized June 2026). Completed work is logged in [CHANGELOG.md](CHANGELOG.md); upcoming work in [ToDo.md](ToDo.md).

**Working mode / database:** the user applies DB changes by pasting the SQL files in [db/](db/) into the Supabase SQL editor (they confirm with screenshots). When you add a feature needing a schema/policy change, write a runnable `.sql` in `db/` and add it to the "Pending Supabase steps" list in [ToDo.md](ToDo.md) rather than assuming it's applied. Catalog `books` writes are open to authenticated users (a documented Phase-1 RLS simplification); `listings`/`listing_photos` are owner-scoped by RLS.

## Prototype Architecture Notes

If working on the existing prototype:

- **Module structure (ES modules, July 4 — split in progress, plan §5.2):** [js/main.js](js/main.js) is the module entry (`<script type="module">`); extracted so far: [js/router.js](js/router.js) (hash routing), [js/api-lookup.js](js/api-lookup.js) (external ISBN lookup + title/author search, no DOM), [js/book-render.js](js/book-render.js) (the §6A contract — its nav actions are injected by main.js via `initBookRender(actions)`, never imported back), and [js/dom-utils.js](js/dom-utils.js) (`escapeHTML`); [js/supabase-config.js](js/supabase-config.js) stays a classic script so `supabaseClient` is a global the modules read. **Rule: any function referenced from HTML** (inline `onclick` in index.html, `onclick` strings in generated markup, or Playwright harness `page.evaluate` calls) **must be listed in the `Object.assign(window, {...})` block at the bottom of main.js** — modules have no global scope, so a missing entry means a silent dead button. When converting an inline handler to `addEventListener`, delete its line from that block. Further extractions (renderers, sell, shelf, scanner…) should go one module per session, harnesses run after each.
- Most JS still lives in [js/main.js](js/main.js). Live data comes from Supabase via `supabaseClient`: browse/search → `loadFeaturedBooks()`/`searchBooks()`; sell → `handleSellBook()` + `ensureBook()`; dashboard tabs → `loadShelfHave()` / `loadShelfWant()` / `loadUserListings()` / `loadProfileSettings()`; ISBN auto-fill → `lookupISBN()` (Edge Function first, then client-side Open Library → Google Books fallback); profile → `viewProfile()` / `toggleFollow()`. User-supplied text is escaped via `escapeHTML()` before `innerHTML`; shelf/profile card click handlers use `addEventListener` (not inline `onclick`) to avoid HTML quoting bugs with special characters in titles.
- **Book rendering goes through one path.** All on-screen book representations route through `renderBook(book, context, density)` (density → `tile | thumb | full`), fed by `normalizeBook(raw)` → `Book` and `bookContext(book)` → `BookContext`, with buttons from `availableActions(book, context)`. **Never hand-build a `.book-card` or duplicate book markup in a new function.** Adding a data point (e.g. a want-count) means extending `BookContext` / `availableActions` — not editing multiple renderers. One cover-fallback constant (`FALLBACK_COVER`); one field name per concept (`coverUrl`, not also `image`/`cover`). The compact picker row in the shelf/sell modals (`renderBookSearchResults`) is deliberately *outside* this contract — it is a selection control, leave it as-is. See [docs/BOOKSHAREZ_ARCHITECTURE.md](docs/BOOKSHAREZ_ARCHITECTURE.md) §6A for the full contract.

  **Consolidation is complete (June 20).** The old renderers (`createBookCard`, `createExternalBookCard`, inline community-shelf card, `renderProfileShelf` DOM loop, `viewListing` DOM manipulation) are deleted. `normalizeBook` → `renderBook` → `_renderTile`/`_renderThumb`/`_renderFull` is the only path.
- **"Pages"** (homepage / dashboard / bookDetail / profilePage) are divs toggled with `style.display`, addressed by **hash routing** (July 4): `#/`, `#/listing/<id>`, `#/book/<bookId>`, `#/profile/<userId>`, `#/dashboard[/<tab>]`. [js/router.js](js/router.js) is the single hash↔page mapping: main.js hands it the page functions via `initRouter(pages)` (no circular import); page functions *record* their route via `setRoute()` (with `replace=true` for within-page changes like dashboard tabs); the `hashchange` listener *applies* routes on back/forward and deep links; a write counter suppresses echo events. The initial route is applied after the first auth state via `applyInitialRoute()` (so `#/dashboard` deep links work with a restored session). External-search books (no catalog id) are deliberately unrouted. When adding a page, add a `setRoute()` call in its show-function, a key in the `initRouter({...})` call, and a case in the router's `_applyRoute()`. Verified by [verify-routing.js](verify-routing.js) (serve the dir on port 7654 first).
- **CSS:** all shared styles live in [css/style.css](css/style.css). Only `#bookCardStyles` and `#bookDetailStyles` are still injected lazily from JS (book grid cards and detail page). Listing card styles (`.listing-card`, `.listing-main`, `.listing-cover`) were moved to `style.css` in June 17 to fix shelf tabs loading before the For Sale tab.
- **API keys in the browser:** `SUPABASE_URL` and `SUPABASE_PUBLISHABLE_KEY` are declared as constants in [js/supabase-config.js](js/supabase-config.js). The Supabase publishable key is safe to be public with RLS enabled. `GOOGLE_BOOKS_API_KEY`, DeepSeek, ISBNdb, and Hardcover keys live only in Supabase Edge Function secrets — never client-side. Client-side Google Books calls are keyless (lower quota, fine for Phase 1).
- **Book detail enrichment (Hardcover):** the detail page is progressively enhanced with community metadata (description, rating, genres, series, page count, category) from the Hardcover GraphQL API. The `book-enrichment` Edge Function ([supabase/functions/book-enrichment/index.ts](supabase/functions/book-enrichment/index.ts)) is cache-first against the `books` table (enrichment columns added by [db/book_enrichment_columns.sql](db/book_enrichment_columns.sql), re-fetched after 30 days), looks up by ISBN-13 with a title+author search fallback, then upserts results. Client side: `runBookEnrichment(isbn, token)` → `fetchBookEnrichment()` → `renderEnrichment()` paints into `#detailEnrichment` (between the ISBN line and the price). It's wired into **both** detail render paths — `_renderFull` (via `viewListing`, the listing page) and `_renderBookPage` (the book page) — and is non-blocking: the page shows immediately, enrichment fills in async, and degrades silently to the old layout when Hardcover has no match. The `currentDetailId` token guards against navigation races. Enrichment CSS lives in [css/style.css](css/style.css) (not injected from JS). Secret: `HARDCOVER_API_TOKEN`.
- **Notifications (rail):** one generic `notifications` table ([db/notifications.sql](db/notifications.sql)) serves all notification types — currently `want_match` (trigger on `listings` insert notifies Want-shelf holders). RLS is owner read/update/delete with **no client INSERT**; rows come only from `SECURITY DEFINER` triggers. Adding a notification type = a new trigger (SQL) + a case in `_renderNotifItem` (client) — never a new table. Header bell UI: `refreshNotifBadge` / `toggleNotifications` / `loadNotifications` / `_openNotification` / `markAllNotificationsRead`; shown/hidden in `applyAuthState`. Degrades silently when the SQL isn't applied. Verified by [verify-notifications.js](verify-notifications.js).
- **Search:** `searchBooks()` runs local DB + `searchBooksAPI()` in parallel. `searchBooksAPI()` tries Google Books first, auto-falls back to Open Library (free, no quota) on any 429/error. Results are merged local-first, paginated 9 at a time with a "View more" button.
- **Sell flow architecture:** selling always routes through the shelf. "Sell Books" header → `showAddToShelfModal('have')` → add to "Books I Have" → "List for Sale" button pre-fills sell modal with `currentListingShelfEntryId` passed through to `listings.shelf_entry_id`.
- **Cover images:** use `object-fit: contain` + `background: #f5f5f5` everywhere (not `cover`) so portrait book covers display without cropping. "For Sale" badge is a `position:absolute` child of a `position:relative` image wrapper.
- The old in-memory `sampleBooks` / `userBooks` arrays have been removed. (`css/style_B.css`, an unlinked stale copy of the stylesheet, was deleted July 4.)
